# Authoritative Facts And Docs Routing

Status: live

Use this map before updating more than one durable harness doc. Repo-owned
docs, tests, policy, and maintainer decisions are authoritative for RunHaven;
HarnessForge output is advisory unless a maintainer promotes it.

## Boundary Types Covered

| Boundary | Owner |
| --- | --- |
| Local repo harness | `AGENTS.md`, `docs/harness/README.md`, this file |
| Generated or owned harness files | `docs/harness/manifest.json` |
| CLI/runtime behavior | `src/runhaven/`, `tests/`, `docs/USAGE.md` |
| Existing project files | Source docs, tests, manifests, and maintainer decisions |
| GitHub Action or CI workflows | `.github/workflows/`, `pins.toml`, `docs/PINNING.md` |
| Optional workflow scaffolds | Not generated by default; maintainer opt-in only |
| Tests and fixtures | `tests/`, `scripts/check_pins.py`, `init.sh` |
| Release/package surface | `pyproject.toml`, `docs/harness/release/`, release evidence |
| Research and source records | `docs/RESEARCH.md`, `docs/harness/research/` |
| Security and privacy | `docs/SECURITY_MODEL.md`, `docs/harness/boundaries/` |
| Platform contracts | macOS 26+, Apple `container` 1.0.0, Python 3.13+ |
| Docs and UX | README, docs index, usage, installation, harness docs |

## Change-To-Docs Routing

| Change | Update First | Usually Skip |
| --- | --- | --- |
| Small internal fix | `progress.md` or no durable doc if state already covers it | README, manifest, roadmap |
| User-facing CLI behavior | `docs/USAGE.md`, tests, relevant help text | Harness research notes |
| Security boundary | `docs/SECURITY_MODEL.md`, `docs/harness/boundaries/security-boundary-map.md`, tests | Marketing copy until behavior is proven |
| Platform/runtime assumption | `README.md`, `docs/INSTALLATION.md`, platform evidence, pins | Unrelated harness state |
| Harness maintenance | This file, focused harness doc, evidence log | Broad repeated prose |
| Release evidence | `docs/harness/release/release-controls.md`, evidence log | Generated templates unless the contract changes |

## Fan-Out Budgets

- Routine change: 0-1 durable doc/state updates.
- User-visible or harness-visible change: 1-3 durable doc/state updates.
- More than 3 durable updates needs an explicit boundary, platform, security,
  release, README, or generated-contract reason.

## Audit And Report Expectations

- Prefer links and compact summaries over duplicated policy prose.
- Keep root instructions as a map, not a full manual.
- Keep machine-local absolute paths, personal tool mandates, and product-local
  HarnessForge research allowlists out of committed target harness docs.
- Treat structural scores as maintenance signals, not real-agent effectiveness.
